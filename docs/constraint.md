# DataFinderAgentOS 开发约束

## 系统背景
- **系统名称**：某政务智能瞭望与智能问数系统
- **系统类型**：Web B/S 结构的数据智能采集和数据智能问数系统
- **架构划分**：用户侧-前台 + 管理侧-后台
- **用户侧定位**：类 ChatGPT 的智能应用，用户通过 Chat 模式与数据直接对话，实现智能问数，通过模型直接获得数据统计、分析、生成图形图像报表
- **用户侧功能**：用户登录、用户注册、用户对话、数字员工、@天气、@新闻、@音乐、@电影、@yummy、报表功能、历史记录、报告导出
- **管理侧功能**：用户管理、功能管理、菜单管理、角色管理、瞭望管理、数据管理、采集管理、数字员工、模型引擎、数智大屏、舆情大屏

## 文档定位
- 本文件用于沉淀当前项目中需要长期遵守的开发约束，作为后续 AI 协作开发的基础上下文。
- 本文件内容以仓库中的真实代码、目录结构和本次学习任务要求为准。

## 环境约束
- 所有开发、运行、测试均在项目根目录下的 `venv/` 虚拟环境中完成。
- Python 启动入口固定为 `run.py`。
- 当前运行端口由 `run.py` 配置为 `10010`。

## 架构约束
- 项目采用轻量 MVC 结构，核心业务代码集中在 `app/`。
- `app/controllers/` 负责路由处理、参数获取、调用模型与模板渲染。
- `app/models/` 负责 SQLite 数据访问与业务相关数据方法。
- `app/templates/` 负责 Tornado 模板页面。
- `app/static/` 负责公共 CSS 与 JS 静态资源。
- 数据库文件固定存放于 `database/finderos.db`。

## 模板分层约束
- `app/templates/admin/` 目录专用于后台管理侧页面（管理侧功能：用户管理、功能管理、菜单管理、角色管理、瞭望管理、数据管理、采集管理、数字员工、模型引擎、数智大屏、舆情大屏）。
- `app/templates/` 下与 `admin/` 同级的模板文件专用于用户侧前台页面（用户侧功能：登录、注册、对话、数字员工调用、报表、历史记录、报告导出）。
- 前后台页面在模板组织、菜单结构、权限控制和路由设计中必须严格区分，不混用页面目录。
- @技能命令（@天气/@新闻/@音乐/@电影/@yummy）作为对话界面的插件式能力，无需独立模板页面，通过数字员工 API 调用实现。

## 路由设计约束
- 后台管理侧路由统一使用 `/admin/` 作为 URL 前缀，对外形式为 `http://xxx/admin/`。
- 用户侧前台路由统一使用根路径体系，对外形式为 `http://xxx/`。
- 后台控制器、后台模板、后台菜单与后台权限设计必须围绕 `/admin/` 路由空间组织。
- 用户侧页面、用户侧对话、报表与历史记录等功能必须围绕根路径路由空间组织。
- 后续新增路由时，前后台不得混用 URL 前缀，不得在用户侧路由空间暴露后台管理页面。
- API路由统一使用 `/api/` 前缀。

## 代码组织约束
- 旧版控制器 `app/controllers/admin.py`（含 AdminUserController 等）为遗留代码，未在 run.py 中注册路由，不应继续使用。
- 新版控制器按模块拆分到独立文件：`admin_user.py`, `admin_role.py`, `admin_function.py`, `admin_menu.py`, `admin_source.py`, `admin_watch.py`, `admin_warehouse.py`, `admin_model.py`, `admin_digital_employee.py`, `admin_profile.py`。
- 数据模型按模块拆分到独立文件：`user.py`, `role.py`, `role_function.py`, `function.py`, `menu.py`, `watch_source.py`, `watch_data.py`, `data_warehouse.py`, `deep_collect.py`, `ai_model.py`, `digital_employee.py`。
- 遗留模板 `user_manage.html` 和空模板 `register.html` 保留待清理或后续实现。

## 界面设计约束
- 系统页面必须支持响应式设计，能够根据不同屏幕宽度调整布局与组件展示。
- 系统页面风格应以沉浸式体验为目标，强调统一视觉语言、主次分明的信息层次和连续的交互体验。
- 系统界面必须支持自适应模式，兼顾移动端与 PC 端访问。
- 前台与后台页面在复用同一套设计语言的同时，应根据业务场景区分信息密度、导航方式和操作复杂度。

## 框架约束
- 当前后端框架为 Tornado。
- 当前数据存储为 SQLite，本地单文件数据库模式。
- 当前认证机制基于 Tornado `secure_cookie` + PBKDF2-HMAC-SHA256 密码哈希。
- 当前 CSRF 防护基于 Tornado XSRF cookie 机制。
- 当前模板引擎使用 Tornado 自带模板语法。
- 前端界面优先使用本地化的 Layui 官方组件包，Bootstrap 作为补充样式和特殊界面需求的支持库。
- 与大模型对话相关的实时通信能力使用 SSE（Server-Sent Events）实现。

## 前端资源约束
- `app/static/dist/` 已放置本地化第三方官方组件资源包。
- 当前已确认包含 `bootstrap-5.3.8-dist` 与 `layui` 两套前端资源。
- 后台管理侧和用户前台侧界面，默认优先基于 Layui 进行页面结构、组件与交互搭建。
- 当 Layui 无法满足响应式样式、栅格补充或特殊页面样式需求时，再引入 Bootstrap 进行补充。
- 页面开发应优先复用 `app/static/dist/` 中的本地资源，避免无必要依赖外部 CDN。

## 安全与配置约束
- 系统安全设计需覆盖 OWASP Top 10 主要风险，并为常见漏洞提供对应安全策略与机制。
- 后续实现中应重点考虑身份认证、权限控制、输入校验、XSS 防护、CSRF 防护、注入防护、敏感配置保护、日志与错误信息控制、依赖安全和安全默认值。
- 所有系统配置文件、环境配置、模型接口配置、安全相关配置与运行参数统一存放在 `config/` 目录下管理。
- 业务代码不得将应抽离的配置常量零散硬编码在控制器、模型或模板中；如需保留临时常量，应在后续迭代中迁移至 `config/`。

## 文档维护约束
- `docs/role.md` 和 `docs/template.md` 视为人类约定文档，默认不由 AI 主动维护。
- `docs/constraint.md` 与 `docs/requirement.md` 由 AI 持续维护。
- `docs/test_case.md` 在收到明确测试相关指令或学习任务要求同步文档时维护。
- 文档中的结构描述必须以真实仓库代码为依据，不能把占位文件当作已完成功能。

## 当前真实状态约束
- 当前已实现的业务模块包括：登录认证、用户管理、角色管理、功能管理、菜单管理、瞭望管理（瞭源+采集）、数据管理（数据仓库+深度采集）、采集管理（瞭源CRUD）、模型引擎、数字员工管理、用户侧登录/注册/对话/数字员工调用/@技能命令/快捷功能，共16个完整模块。
- 后台管理侧模块均基于 Layui 实现前端界面，支持分页搜索、CRUD操作；用户侧前台基于独立浅色系政务简约风格（chat.css）。
- 深度采集模块集成了 crawl4ai 网页爬虫和数字员工-采集专员调度机制，支持异步任务执行和状态监控。
- 模型引擎支持 SSE 流式对话（代理到 OpenAI 兼容 API），数字员工支持 LLM 型和 API 型两种模式。
- RBAC 权限体系基于 用户-角色-功能 三级模型，通过 role_functions 表实现功能权限分配，通过 menus 表实现菜单可见性控制。
- `register.html` 已实现为完整注册页面；`app/templates/admin/user_manage.html` 为遗留文件，不应视为已实现模块。
- `app/controllers/admin.py` 中定义的旧版控制器未在 run.py 中注册路由，属于未使用的遗留代码。
- WebSocket 和 OpenAI 原生 SDK 尚未接入，当前流式对话通过 Tornado AsyncHTTPClient + SSE 实现代理转发。
- 用户侧前台功能中，用户登录、注册、对话、数字员工调用、@技能命令已实现；报表、独立历史记录页面、报告导出和管理侧大屏（数智大屏、舆情大屏）尚未实现。

## 学习阶段结论
- 后续分析、实现和测试任务应优先围绕瞭望采集、数据仓库、深度采集、模型引擎、数字员工等已实现模块的增强和 bug 修复展开。
- 涉及 AI 对话能力的扩展应优先围绕已有的 SSE 代理模式进行。
- 用户侧前台对话界面（chat.html）已实现，支持五区布局、SSE流式、@数字员工、/快捷命令。
- 日志审计、独立报表页面和报告导出是后续应优先补充的新模块。
- 后续页面实现必须同时满足响应式、沉浸式和移动端/PC 端自适应要求。
- 后续配置治理和安全治理必须从项目早期开始纳入设计，而不是等功能完成后补做。
